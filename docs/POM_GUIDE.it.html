Questa guida è esplicativa

Questa pagina aiuta chi legge a capire POM. Non è il regolamento operativo. Le regole canoniche restano in README.md, AGENTS.MD, prompts/, skills/ e templates/.

Quando POM cambia, questa guida va riletta e riallineata perché continui a essere utile a chi sta imparando il metodo.

Cos'è POM

POM significa Project Operating Memory. È un metodo riusabile per conservare il minimo contesto affidabile necessario a capire dove si trova un progetto, perché si trova lì e qual è il prossimo passo sicuro.

Un passo sicuro è un'azione che il prossimo lettore o agente può compiere senza dover ricostruire il contesto, ripetere decisioni già prese o modificare parti stabili del progetto alla cieca.

POM collega codice sorgente, decisioni, specifiche, piani di lavoro, pagine wiki, mockup, test e istruzioni per agenti. Lo scopo non è documentare tutto. Lo scopo è rendere il progetto riprendibile dopo una pausa, un passaggio di mano o un cambio di agente.

Perché esiste

Continuità

I progetti perdono contesto

Molti ragionamenti importanti restano in chat, memoria o nella testa di qualcuno. POM porta il contesto durevole nel repository.

Verifica

Fatto richiede evidenza

POM tratta il completamento come una tesi da verificare, non come una casella da fidarsi a occhi chiusi.

Orientamento

Gli agenti devono leggere le fonti giuste

Domande diverse appartengono a fonti diverse. POM aiuta l'agente a leggere il file corretto prima di agire.

Cosa non è POM

Non è un project manager

POM non sostituisce issue tracker, roadmap tool o rituali di team. Mantiene la memoria operativa vicino al lavoro.

Non è un framework runtime

POM non deve imporre l'architettura applicativa. La struttura esistente del progetto viene mappata prima di proporre migrazioni.

Non è un archivio di documenti

Avere più file non è automaticamente meglio. La memoria deve essere utile, corrente e collegata a una fonte autorevole.

Non sostituisce Git

Git resta la memoria storica dettagliata. POM conserva la sintesi operativa corrente.

Principi fondanti

1. Memoria operativa

Conservare il minimo contesto affidabile che consente di riprendere il lavoro in sicurezza. Evitare log, archivi e documentazione decorativa.

2. Verifica

Una specifica, un task o una decisione non sono completi finché l'obiettivo dichiarato non è davvero soddisfatto e l'evidenza lo dimostra.

3. Organizzazione

La struttura aiuta solo quando protegge memoria e verifica. L'organizzazione serve il lavoro, non il contrario.

Prima installazione e aggiornamenti

Prima installazione

Dalla root del progetto target, scaricare lo script di bootstrap ed eseguirlo. L'installer interattivo permette di scegliere un profilo come minimal, wiki, decisions, full o adopt.

curl -fsSL https://raw.githubusercontent.com/FabioMalpezzi/pom/main/bootstrap-pom.mjs -o bootstrap-pom.mjs node bootstrap-pom.mjs

Dopo che il bootstrap ha installato pom/, usare Read pom/skills/seed.md and set up POM for this project. Su un repository esistente, usare Read pom/skills/adopt.md and adopt POM without changing the existing structure.

Aggiornare POM già installato

Quando un progetto ha già pom/, usare la skill sync: l'agente controlla modifiche locali, aggiorna pom/, rigenera le sezioni, esegue il lint e prepara solo i file collegati.

Read pom/skills/sync.md and refresh this project's POM installation.

npm run pom:update

pom:update funziona con installazioni Git-managed e copie vendorizzate pulite. Si ferma se pom/ contiene modifiche locali e suggerisce la skill sync invece di sovrascriverle.

Se un progetto più vecchio non ha ancora pom:update, eseguire una volta node bootstrap-pom.mjs --profile refresh per installare l'updater corrente.

Usare POM su un nuovo progetto

  1. Partire piccoli. Scegliere il profilo di adozione più leggero che dia al progetto un punto di ripartenza affidabile.
  2. Creare la memoria di base: istruzioni per agenti, configurazione e stato di progetto.
  3. Aggiungere wiki, decisioni, task plan o test strutturati solo quando il progetto ne ha bisogno.
  4. Lasciare a Git la storia dettagliata. Lasciare a POM la sintesi corrente.
Prima richiesta utile: Read pom/skills/seed.md and set up POM for this project.

Usare POM su un progetto esistente

In un repository esistente, POM deve prima osservare e mappare. Non deve riorganizzare il progetto solo per farlo assomigliare a un esempio canonico.

  1. Leggere documentazione esistente, radici del sorgente, test, decisioni e convenzioni.
  2. Registrare la struttura reale in pom.config.json.
  3. Individuare il più piccolo vuoto di memoria che danneggia ripartenza, verifica o fonte autorevole.
  4. Adottare in modo incrementale. Le migrazioni sono decisioni esplicite, non effetti collaterali dell'installazione.
Prima richiesta utile: Read pom/skills/adopt.md and adopt POM without changing the existing structure.

Overlay su progetti esterni

Quando il repository è clonato da un upstream che non controlliamo, POM deve comportarsi come overlay locale di comprensione, non come governance del progetto.

  • Non imporre convenzioni POM a docs/, tests/, ADR, struttura sorgente, processo di release o contenuto delle pull request upstream.
  • Usare la memoria locale per mappare architettura, entrypoint, moduli, test, convenzioni, rischi e domande aperte.
  • Tenere gli artefatti overlay fuori dai contributi upstream, salvo richiesta esplicita del progetto.
  • Prima di una PR, stageare solo la modifica reale per upstream ed eseguire i controlli propri del progetto.
Modalità documentata: Read pom/specs/SPEC-0004-external-project-overlay.md and use POM as a local understanding overlay.

Livelli di utilizzo

Minimal

Memoria di ripartenza

Usa POM per istruzioni agenti, configurazione e stato di progetto. È adatto quando il progetto è piccolo o ancora in formazione.

Light

Memoria mirata

Aggiunge wiki o decisioni quando risolvono un vero problema di coordinamento. Può corrispondere a profili come wiki o decisions.

Full

Memoria governata

Usa wiki, decisioni, pianificazione, verifica, lint e regole di handoff più forti per progetti grandi o di lunga durata.

L'installer supporta anche profili pratici come adopt, refresh e custom. Il profilo è un punto di partenza, non una medaglia di maturità.

Sperimentazioni isolate

Gli esperimenti sono preziosi, ma non devono contaminare il sorgente stabile prima della valutazione. POM li tratta come lavoro temporaneo finché non vengono consolidati deliberatamente.

  • Usare un branch o una Git worktree come exp/<topic> per lavori ampi o rischiosi.
  • Usare /tmp o experiments/<topic> per prove brevi ed evidenza eliminabile.
  • Tenere isolati dipendenze sperimentali, file di ambiente, output generati e repository esterni.
  • Promuovere solo l'apprendimento selezionato: scartare, sintetizzare, scrivere una decisione, creare un task plan o reimplementare pulito.

Git e POM

Git è la memoria storica fine. Registra cosa è cambiato e quando. POM è la sintesi operativa. Spiega cosa conta adesso, perché conta e da dove riprendere.

Non trasformare i documenti POM in changelog manuali. Se serve il dettaglio storico, si legge Git. Se serve il contesto di ripartenza, si legge POM.

Raccomandazioni finali

  • Partire dal profilo POM più piccolo che sia davvero utile.
  • Creare memoria solo quando aiuta il lavoro futuro a ripartire, verificare o non riscoprire informazioni.
  • Leggere le fonti correnti prima di fare affermazioni sul progetto.
  • Usare Git per la storia e POM per la sintesi operativa corrente.
  • Tenere isolati gli esperimenti finché il loro apprendimento non viene selezionato per la promozione.
  • Potare le regole che aggiungono processo senza proteggere la memoria.
  • Quando si usa o si evolve POM stesso, chiedersi cosa servirà al prossimo lettore o agente per compiere il prossimo passo sicuro nel progetto.
