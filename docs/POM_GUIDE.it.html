Cos'è POM

POM significa Project Operating Memory. In pratica, è il piccolo insieme di note di progetto che spiega che cosa conta adesso: che cosa il progetto sta cercando di fare, che cosa è già stato deciso, che cosa è ancora in discussione e dove la prossima persona o il prossimo agente AI deve guardare prima di agire.

Serve perché i progetti perdono contesto in fretta. Un'idea utile può restare in una chat, una motivazione tecnica può rimanere nella memoria di qualcuno e una domanda aperta può non essere visibile a chi arriva dopo. POM dà a queste conoscenze un posto stabile dentro il repository.

Il metodo è volutamente leggero. Non deve obbligare ogni progetto a usare le stesse cartelle e non deve creare documenti solo perché esiste un template. Aggiunge solo la memoria che rende più chiaro e più sicuro il prossimo passo.

Questa guida è solo un'introduzione

Questa pagina spiega POM con un linguaggio diretto, per chi lo incontra per la prima volta. Non è il regolamento che gli agenti devono seguire mentre lavorano. Le regole operative vivono in README.md, AGENTS.MD, prompts/, skills/ e templates/.

Il POM Wiki Reader generato è una copia comoda da leggere nel browser della wiki radice. La wiki serve alla memoria corrente del progetto; questa pagina serve a capire che cos'è POM e perché esistono i suoi pezzi.

In un progetto target, la wiki viene aggiunta solo quando il progetto ha conoscenze che devono sopravvivere alla chat corrente. La fonte è il Markdown sotto wiki/. Il reader HTML è solo una vista comoda e può essere rigenerato.

Quando POM cambia, questa guida va riletta e riallineata perché continui ad aiutare chi sta imparando il metodo.

Perché esiste

Continuità

I progetti perdono contesto

Molti ragionamenti importanti restano in chat, memoria o nella testa di qualcuno. POM porta nei file di progetto solo le parti che vale la pena conservare.

Verifica

Fatto richiede evidenza

Quando un task, una specifica o una decisione vengono dichiarati completi, POM chiede che cosa lo dimostra: test, documenti controllati, comportamento funzionante o un'altra verifica chiara.

Orientamento

Gli agenti devono leggere le fonti giuste

Codice, appunti, decisioni, specifiche e wiki rispondono a domande diverse. POM aiuta l'agente a leggere il file che può davvero rispondere alla domanda prima di cambiare qualcosa.

Cosa non è POM

Non è un project manager

POM non sostituisce issue tracker, roadmap tool o rituali di team. Mantiene la memoria operativa vicino al lavoro.

Non è un framework runtime

POM non deve imporre l'architettura applicativa. La struttura esistente del progetto viene mappata prima di proporre migrazioni.

Non è un archivio di documenti

Avere più file non è automaticamente meglio. Un documento POM deve aiutare qualcuno ad agire, e deve essere chiaro quale fonte del progetto lo sostiene.

Non sostituisce Git

Git resta la memoria storica dettagliata. POM conserva la sintesi operativa corrente.

Principi fondanti

1. Memoria operativa

Conservare solo il contesto affidabile che consente di riprendere il lavoro in sicurezza. POM non è il diario di tutto quello che è successo; è la sintesi utile per capire che cosa fare dopo.

2. Verifica

Una specifica, un task o una decisione non sono completi solo perché sono stati scritti. Sono completi quando l'obiettivo dichiarato è davvero soddisfatto e l'evidenza lo dimostra.

3. Organizzazione

Cartelle e template aiutano solo quando rendono più chiaro il prossimo passo. POM deve seguire la forma del progetto prima di chiedere al progetto di cambiare forma.

Strumenti di POM

I nomi qui sotto sono il vocabolario del metodo. Sono pezzi volutamente piccoli: ognuno risponde a una domanda pratica diversa.

Strumento Scopo File tipico
Wiki persistente Memoria mantenuta del progetto: che cosa si sa adesso, perché conta e da dove arriva quella conoscenza. wiki/
Open Discussion Spazio per desideri, ipotesi, alternative e domande aperte mentre sono ancora in discussione. Non è ancora una decisione o un requisito. analysis/ o percorso approvato dal progetto
Project State Nota breve di passaggio: posizione corrente, lavoro recente rilevante e prossime azioni sicure. PROJECT_STATE.md
ADR e spec Gli ADR sono registri di decisione: spiegano che cosa è stato scelto e perché. Le specifiche descrivono il comportamento che il progetto si aspetta. decisions/, specs/
Task plan Scomposizione del lavoro che dice che cosa deve succedere e come verrà controllato il completamento. tasks/
Config Dice a POM dove questo progetto tiene codice, documenti, test, decisioni, file generati e altri percorsi importanti. pom.config.json
Artifact Policy Dice all'agente se un file si può modificare, richiede approvazione, è generato o deve restare storico. pom.config.json
Lint Esegue controlli ripetibili per regole che non dovrebbero dipendere dalla memoria o dall'interpretazione. npm run pom:lint
Skill Punti di ingresso brevi che dicono all'agente che tipo di lavoro sta facendo e quali istruzioni leggere. skills/
Prompt Procedure passo passo per applicare il metodo in situazioni ricorrenti. prompts/
Template Forme riusabili per mantenere leggibili note, piani e decisioni simili tra loro. templates/

Come funziona in pratica

POM si può leggere come un ciclo semplice: capire la richiesta, scegliere il modo giusto di lavorare, scrivere solo memoria utile e verificare le affermazioni importanti prima di chiudere il lavoro.

1. Richiesta

L'utente dichiara il lavoro e il risultato atteso

Esempio: "Trasforma la specifica sul recupero password in un piano verificabile".

2. Workflow

L'agente sceglie la procedura

L'agente sceglie una skill breve, legge le istruzioni collegate e usa un template solo quando serve davvero un documento.

3. Memoria

Il risultato entra nel repository

Se il risultato servirà più avanti, diventa un documento del progetto invece di restare sepolto nella chat.

4. Verifica

POM controlla l'affidabilità

POM controlla che il risultato sia affidabile. Per il codice di solito significa test. Per i documenti significa controllare le fonti, cercare contraddizioni e sfidare la conclusione prima di considerarla chiusa.

Prima installazione e aggiornamenti

Prima installazione

Dalla root del progetto target, scaricare lo script di bootstrap ed eseguirlo con un preset esplicito. POM non deve indovinare se il repository è tuo, condiviso con un team o di un upstream esterno.

curl -fsSL https://raw.githubusercontent.com/FabioMalpezzi/pom/main/bootstrap-pom.mjs -o bootstrap-pom.mjs node bootstrap-pom.mjs --preset owned

Scegliere il preset in base al rapporto con il repository:

  • owned: progetto tuo; POM può aiutare a governare metodo e struttura.
  • team: progetto condiviso; POM deve preservare le convenzioni del team.
  • overlay: repository di terzi; POM resta memoria locale di comprensione.
  • minimal: setup locale minimo, senza assumere ownership.

Per un nuovo progetto, dopo l'installazione chiedere all'agente:

Leggi pom/skills/seed.md e configura POM per questo progetto.

Per un repository esistente, chiedere invece:

Leggi pom/skills/adopt.md e adotta POM senza cambiare la struttura esistente.

Per forzare la lingua dei messaggi CLI, usare --lang:

node bootstrap-pom.mjs --preset team --lang it npm run pom:help -- --lang it

Aggiornare POM già installato

Quando un progetto ha già pom/, usare la skill sync: l'agente controlla modifiche locali, aggiorna pom/, rigenera le sezioni, esegue il lint e prepara solo i file collegati.

Leggi pom/skills/sync.md e aggiorna l'installazione POM di questo progetto.

npm run pom:update

pom:update funziona con installazioni Git-managed e copie vendorizzate pulite. Si ferma se pom/ contiene modifiche locali e suggerisce la skill sync invece di sovrascriverle. Non cambia modalità di adozione: per quello usare pom:init -- --preset ....

Se un progetto più vecchio non ha ancora pom:update, eseguire una volta node bootstrap-pom.mjs --profile refresh per installare l'updater corrente.

Usare POM su un nuovo progetto

  1. Partire piccoli. Scegliere solo la quantità di POM che dà al progetto un punto di ripartenza affidabile.
  2. Creare la memoria di base: istruzioni per agenti, file di configurazione e breve stato di progetto.
  3. Non creare cartelle applicative, decisioni formali (ADR) o specifiche prima che la direzione del progetto sia esplicita. Usare Open Discussion per alternative non risolte.
  4. Aggiungere pagine wiki, decisioni, task plan o test strutturati solo quando il progetto ne ha bisogno.
  5. Lasciare a Git la storia dettagliata. Lasciare a POM la sintesi corrente.
Prima richiesta utile: Leggi pom/skills/seed.md e configura POM per questo progetto.

Usare POM su un progetto esistente

In un repository esistente, POM deve prima osservare e mappare come il progetto lavora già. Non deve riorganizzare il progetto solo per farlo assomigliare a un esempio.

  1. Leggere documentazione esistente, radici del sorgente, test, decisioni e convenzioni.
  2. Registrare la struttura reale in pom.config.json.
  3. Registrare l'Artifact Policy: quali file sono generati, quali sono storici e quali richiedono approvazione esplicita prima della modifica.
  4. Individuare il più piccolo vuoto di memoria che rende più difficile ripartire, verificare o restare allineati alle fonti del progetto.
  5. Adottare in modo incrementale. Le migrazioni sono decisioni esplicite, non effetti collaterali dell'installazione.
owned significa che POM può aiutare a definire le regole del progetto. team significa che POM deve preservare le convenzioni condivise salvo cambio esplicito del team.
Prima richiesta utile: Leggi pom/skills/adopt.md e adotta POM senza cambiare la struttura esistente.

Overlay su progetti esterni

A volte il repository appartiene a un altro progetto, e POM serve solo ad aiutare chi lavora localmente a capirlo. In quel caso POM deve comportarsi come uno strato privato di note, non come regole per il progetto originale upstream.

  • Non imporre convenzioni POM a docs/, tests/, ADR, struttura sorgente, processo di release o contenuto delle pull request upstream.
  • Usare la memoria locale per mappare architettura, entrypoint, moduli, test, convenzioni, rischi e domande aperte.
  • Tenere le note POM locali fuori dai contributi upstream, salvo richiesta esplicita del progetto.
  • Tenere il lavoro overlay su un suo branch o worktree; non fare merge di quel branch nella branch di contributo.
  • Prima di una PR, trasferire solo modifiche selezionate non-POM, stageare la modifica reale per upstream ed eseguire i controlli propri del progetto.
node bootstrap-pom.mjs --preset overlay npm run pom:init -- --preset overlay
Modalità documentata: Read pom/specs/SPEC-0004-external-project-overlay.md and use POM as a local understanding layer.

Livelli di utilizzo

Minimal

Memoria di ripartenza

Usa POM per istruzioni agenti, configurazione e stato di progetto. È sufficiente quando il progetto è piccolo o ancora in formazione.

Light

Memoria mirata

Aggiunge wiki o decisioni solo quando risolvono un vero problema di coordinamento. Può corrispondere a profili come wiki o decisions.

Full

Memoria più strutturata

Usa wiki, decisioni, pianificazione, verifica, controlli automatici e regole di passaggio più forti per progetti grandi o di lunga durata.

L'installer supporta anche profili pratici come adopt, refresh e custom. Il profilo è un punto di partenza, non una medaglia di maturità.

Sperimentazioni isolate

Gli esperimenti sono preziosi, ma non devono cambiare il progetto stabile prima della valutazione. POM li tratta come lavoro temporaneo finché qualcuno sceglie deliberatamente che cosa tenere.

  • Usare un branch o una Git worktree come exp/<topic> per lavori ampi o rischiosi.
  • Usare /tmp o experiments/<topic> per prove brevi ed evidenza eliminabile.
  • Tenere isolati dipendenze sperimentali, file di ambiente, output generati e repository esterni.
  • Promuovere solo l'apprendimento selezionato: scartare, sintetizzare, scrivere una decisione, creare un task plan o reimplementare pulito.

Git e POM

Git è la memoria storica dettagliata. Registra cosa è cambiato e quando. POM è la sintesi operativa corrente. Spiega cosa conta adesso, perché conta e da dove riprendere.

Non trasformare i documenti POM in changelog manuali. Se serve il dettaglio storico, si legge Git. Se serve il contesto di ripartenza, si legge POM.

Raccomandazioni finali

  • Partire dal profilo POM più piccolo che sia davvero utile.
  • Creare memoria solo quando aiuta il lavoro futuro a ripartire, verificare o non riscoprire informazioni.
  • Leggere le fonti correnti prima di fare affermazioni sul progetto.
  • Tenere desiderata e alternative non risolte in Open Discussion finché non vengono promosse a spec, decisione, wiki o task plan.
  • Controllare l'Artifact Policy prima di modificare file con regole speciali.
  • Usare Git per la storia e POM per la sintesi operativa corrente.
  • Tenere isolati gli esperimenti finché il loro apprendimento non viene selezionato per la promozione.
  • Togliere le regole che aggiungono processo senza proteggere la memoria.
  • Quando si usa o si evolve POM stesso, chiedersi cosa servirà al prossimo lettore o agente per compiere il prossimo passo sicuro nel progetto.
