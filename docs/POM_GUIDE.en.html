This guide is explanatory

This page helps a reader understand POM. It is not the operational rulebook. The canonical operating rules remain in README.md, AGENTS.MD, prompts/, skills/, and templates/.

When POM changes, this guide should be reviewed so it remains useful to humans who are learning the method.

What POM is

POM stands for Project Operating Memory. It is a reusable method for preserving the minimum reliable context needed to understand where a project is, why it is there, and what the next safe step is.

POM connects source code, decisions, specifications, task plans, wiki pages, mockups, tests, and agent instructions. The point is not to document everything. The point is to make the project restartable after a pause, a handoff, or a change of agent.

Why it exists

Continuity

Projects lose context

Important reasoning often lives in chat, memory, or a person's head. POM moves durable context into the repository.

Verification

Done needs evidence

POM treats completion as a claim to verify, not as a checkbox to trust.

Navigation

Agents need source authority

Different questions belong to different sources. POM helps the agent read the right file before acting.

What POM is not

Not a project manager

POM does not replace issue trackers, roadmap tools, or team rituals. It keeps operating memory close to the work.

Not a runtime framework

POM should not impose application architecture. Existing project structure is mapped before any migration is proposed.

Not a document dump

More files are not automatically better. Memory must be useful, current, and connected to source authority.

Not a substitute for Git

Git remains the detailed historical memory. POM captures the current operating synthesis.

Founding principles

1. Operating memory

Keep the minimum reliable context that lets someone resume work safely. Avoid logs, archives, and decorative documentation.

2. Verification

A spec, task, or decision is not complete until its stated goal is actually met and the evidence supports that claim.

3. Organization

Structure helps only when it protects memory and verification. Organization serves the work, not the other way around.

Using POM on a new project

  1. Start small. Choose the lightest adoption profile that gives the project a reliable restart point.
  2. Create the basic memory: agent instructions, config, and project state.
  3. Add wiki, decisions, task plans, or structured tests only when the project needs them.
  4. Let Git carry detailed history. Let POM carry the current synthesis.
Good first request: Read pom/skills/seed.md and set up POM for this project.

Using POM on an existing project

For an existing repository, POM should first observe and map. It should not reorganize the project just to fit a canonical example.

  1. Read existing documentation, source roots, tests, decisions, and conventions.
  2. Record the actual structure in pom.config.json.
  3. Identify the smallest memory gap that hurts restart, verification, or source authority.
  4. Adopt incrementally. Migrations are explicit decisions, not installation side effects.
Good first request: Read pom/skills/adopt.md and adopt POM without changing the existing structure.

Levels of use

Minimal

Restart memory

Use POM for agent instructions, config, and project state. Best when the project is small or still forming.

Light

Focused memory

Add the wiki or decision records where they solve a real coordination problem. This maps to profiles such as wiki or decisions.

Full

Governed memory

Use wiki, decisions, planning, verification, lint, and stronger handoff rules for larger or long-running projects.

The installer also supports practical profiles such as adopt, refresh, and custom. The profile is a starting point, not a badge of maturity.

Isolated experiments

Experiments are valuable, but they should not contaminate stable source before evaluation. POM treats experiments as temporary work until they are deliberately consolidated.

  • Use a branch or Git worktree such as exp/<topic> for broad or risky work.
  • Use /tmp or experiments/<topic> for short trials and disposable evidence.
  • Keep trial dependencies, environment files, generated output, and external repositories isolated.
  • Promote only selected learning: discard, synthesize, write a decision, create a task plan, or reimplement cleanly.

Git and POM

Git is the fine-grained historical memory. It records what changed and when. POM is the operating synthesis. It explains what matters now, why it matters, and where to resume.

Do not turn POM documents into manual changelogs. If historical detail is needed, read Git. If restart context is needed, read POM.

Final recommendations

  • Start with the smallest useful POM profile.
  • Create memory only when it helps future work restart, verify, or avoid rediscovery.
  • Read current sources before making claims about the project.
  • Use Git for history and POM for current operating synthesis.
  • Keep experiments isolated until their learning is selected for promotion.
  • Prune rules that add process without protecting memory.
  • When using or evolving POM itself, ask what a future reader or agent will need to know to take the next safe step in the project.
