What POM is

POM stands for Project Operating Memory. In practice, it is the small set of project notes that explains what matters now: what the project is trying to do, what has already been decided, what is still being discussed, and where the next person or AI agent should look before acting.

It exists because projects lose context quickly. A useful idea may stay in a chat thread, a design reason may remain in someone's memory, and an open question may never become visible to the next person. POM gives those pieces of knowledge a stable place inside the repository.

The method is intentionally lightweight. It should not force every project into the same folder structure, and it should not create documents just because a template exists. It adds only the memory that helps the next step be clearer and safer.

This guide is only an introduction

This page explains POM in plain language for someone who is seeing it for the first time. It is not the rulebook that agents must follow while they work. The operational rules live in README.md, AGENTS.MD, prompts/, skills/, and templates/.

The generated POM Wiki Reader is a browser-friendly copy of the root wiki. The wiki is for current project memory; this page is for understanding what POM is and why the pieces exist.

In a target project, the wiki is added only when the project needs knowledge that should survive beyond the current chat. Its source is Markdown under wiki/. The HTML reader is only a convenient view and can be regenerated.

When POM changes, this guide should be reviewed so it continues to help people who are learning the method.

Why it exists

Continuity

Projects lose context

Important reasoning often lives in chat, memory, or a person's head. POM moves only the parts worth keeping into project files.

Verification

Done needs evidence

When a task, specification, or decision is called complete, POM asks what proves that claim: tests, reviewed documents, working behavior, or another clear check.

Navigation

Agents need the right source

Code, notes, decisions, specifications, and wiki pages answer different questions. POM helps the agent read the file that can actually answer the question before it changes anything.

What POM is not

Not a project manager

POM does not replace issue trackers, roadmap tools, or team rituals. It keeps operating memory close to the work.

Not a runtime framework

POM should not impose application architecture. Existing project structure is mapped before any migration is proposed.

Not a document dump

More files are not automatically better. A POM document should help someone act, and it should be clear which project source supports it.

Not a substitute for Git

Git remains the detailed historical memory. POM captures the current operating synthesis.

Founding principles

1. Operating memory

Keep just enough reliable context for someone to restart safely. POM is not a diary of everything that happened; it is the useful summary for what to do next.

2. Verification

A specification, task, or decision is not complete just because it was written down. It is complete when the stated goal is actually met and the evidence supports that claim.

3. Organization

Folders and templates help only when they make the next step clearer. POM should follow the shape of the project before it asks the project to change shape.

POM tools

The names below are the vocabulary of the method. They are deliberately small pieces: each one answers a different practical question.

Tool Purpose Typical file
Persistent Wiki A maintained project memory: what is known now, why it matters, and where the knowledge came from. wiki/
Open Discussion A place for wishes, hypotheses, alternatives, and open questions while they are still being discussed. It is not yet a decision or a requirement. analysis/ or project-approved path
Project State A short handoff note: current position, recent relevant work, and the next safe actions. PROJECT_STATE.md
ADR and Spec ADRs are decision records: they explain what was chosen and why. Specifications describe the behavior the project expects. decisions/, specs/
Task Plan A work breakdown that says what must happen and how completion will be checked. tasks/
Config Tells POM where this project keeps source code, docs, tests, decisions, generated files, and other important paths. pom.config.json
Artifact Policy Tells the agent whether a file can be edited, needs approval, is generated, or should stay historical. pom.config.json
Lint Runs repeatable checks for rules that should not depend on memory or interpretation. npm run pom:lint
Skills Short entry points that tell an agent which kind of work it is doing and which instructions to read. skills/
Prompts Step-by-step procedures for applying the method in recurring situations. prompts/
Templates Reusable document shapes so similar notes, plans, and decisions stay easy to read. templates/

POM Project Reader

The POM Project Reader is an experimental local web server for reading a project in the browser. It is a live consultation surface over repository files, not a new source of authority.

Launch it from the project root

By default the server uses the current working directory as the project root and port 4173. In the POM Source repository, run:

node experiments/wiki-agent-orchestration/mini-ui/server.mjs --port 4173

In a target project where POM is installed under pom/, run:

node pom/experiments/wiki-agent-orchestration/mini-ui/server.mjs --port 4173

Use --root or --dir, --port, and --annotations-dir when those paths must be explicit.

node pom/experiments/wiki-agent-orchestration/mini-ui/server.mjs --port 4173 --root . --annotations-dir .pom-reader/annotations

Then open http://127.0.0.1:4173.

Read, search, annotate

The reader starts from wiki/index.md when a wiki exists. Without a wiki, it shows a generated project entry page and still exposes project documentation and source files.

When pom.config.json exists, configured roots refine the document set and classification: documentation roots, decision records, task plans, analysis, source roots, tests, mockup packages, root-level Markdown, and generated-output exclusions. Without the config, the reader keeps its built-in allowlist. The UI shows which mode is active.

It supports thematic and tree navigation, pinned or collapsible side panels, project-wide rg search when ripgrep is installed, search inside the open file, optional regex mode, Markdown tables, fixed-width text, syntax-highlighted code, and useful source line numbers. Rendering rejects files above 1 MB and binary-looking files; search skips files above 1 MB. The document pane keeps prose readable while giving code blocks and tables more room on large screens.

Annotations are JSON work files under the configured annotation directory. The default is experiments/wiki-agent-orchestration/evidence/annotations/. Treat them as runtime evidence unless the project chooses to archive one. The right panel separates new notes, in-progress work, and processed history; the UI can reopen the target document when it still exists. It does not talk directly to an AI agent, and browser-based source editing is not part of the current workflow. The server binds to 127.0.0.1 and sends restrictive browser security headers.

node experiments/wiki-agent-orchestration/wiki-tools.mjs claim-next --by codex

How POM works in practice

POM can be read as a simple loop: understand the request, choose the right way of working, write only useful memory, and verify important claims before closing the work.

1. Request

The user states the work and expected result

Example: "Turn the password recovery spec into a verifiable plan".

2. Workflow

The agent chooses the procedure

The agent chooses a short skill, reads the linked instructions, and uses a template only when a real document is needed.

3. Memory

The result enters the repository

If the result matters later, it becomes a project document instead of staying buried in chat.

4. Verification

POM checks reliability

POM checks that the result is reliable. For code this usually means tests. For documents it means checking the sources, looking for contradictions, and challenging the conclusion before treating it as settled.

First install and updates

First install

From the target project root, download the bootstrap script and run it with an explicit preset. POM does not guess whether the repository is yours, shared with a team, or an external upstream.

curl -fsSL https://raw.githubusercontent.com/FabioMalpezzi/pom/main/bootstrap-pom.mjs -o bootstrap-pom.mjs node bootstrap-pom.mjs --preset owned

Choose the preset from your relationship to the repository:

  • owned: project is yours; POM may help govern method and structure.
  • team: shared project; POM must preserve team conventions.
  • overlay: third-party repository; POM stays local understanding memory.
  • minimal: smallest local setup, with no ownership assumption.

For a new project, after installation ask the agent:

Read pom/skills/seed.md and set up POM for this project.

For an existing repository, ask instead:

Read pom/skills/adopt.md and adopt POM without changing the existing structure.

To force CLI message language, use --lang:

node bootstrap-pom.mjs --preset team --lang en npm run pom:help -- --lang en

Update installed POM

When a project already has pom/, use the sync skill so the agent checks local changes, updates pom/, refreshes generated sections, runs lint, and stages only related files.

Read pom/skills/sync.md and refresh this project's POM installation.

npm run pom:update

pom:update works with Git-managed installs and clean vendored copies. It stops if pom/ has local changes and suggests the sync skill instead of overwriting them. It does not change adoption mode; use pom:init -- --preset ... for that.

If an older project does not have pom:update yet, run node bootstrap-pom.mjs --profile refresh once to install the current updater.

Using POM on a new project

  1. Start small. Choose only the amount of POM that gives the project a reliable restart point.
  2. Create the basic memory: instructions for agents, a configuration file, and a short project state.
  3. Do not create application folders, decision records (ADRs), or specifications before the project direction is explicit. Use Open Discussion for unresolved alternatives.
  4. Add wiki pages, decisions, task plans, or structured tests only when the project needs them.
  5. Let Git carry detailed history. Let POM carry the current synthesis.
Good first request: Read pom/skills/seed.md and set up POM for this project.

Using POM on an existing project

For an existing repository, POM should first observe and map how the project already works. It should not reorganize the project just to make it resemble an example.

  1. Read existing documentation, source roots, tests, decisions, and conventions.
  2. Record the actual structure in pom.config.json.
  3. Record the Artifact Policy: which files are generated, which are historical, and which need explicit approval before editing.
  4. Identify the smallest memory gap that makes restart, verification, or alignment with project sources harder.
  5. Adopt incrementally. Migrations are explicit decisions, not installation side effects.
owned means POM may help define project rules. team means POM must preserve shared conventions unless the team explicitly changes them.
Good first request: Read pom/skills/adopt.md and adopt POM without changing the existing structure.

Overlay on external projects

Sometimes the repository belongs to another project, and POM is only there to help you understand it locally. In that case, POM should act as a private layer of notes, not as rules for the upstream project.

  • Do not impose POM conventions on upstream docs/, tests/, ADRs, source layout, release process, or pull-request contents.
  • Use local memory to map architecture, entrypoints, modules, tests, conventions, risks, and open questions.
  • Keep local POM notes out of upstream contributions unless the project explicitly wants them.
  • Keep overlay work on its own branch or worktree; do not merge that branch into the contribution branch.
  • Before a PR, transfer only selected non-POM changes, stage the real upstream change, and run the upstream project's own checks.
node bootstrap-pom.mjs --preset overlay npm run pom:init -- --preset overlay
Documented mode: Read pom/specs/SPEC-0004-external-project-overlay.md and use POM as a local understanding layer.

Levels of use

Minimal

Restart memory

Use POM for agent instructions, configuration, and project state. This is enough when the project is small or still forming.

Light

Focused memory

Add the wiki or decision records only where they solve a real coordination problem. This maps to profiles such as wiki or decisions.

Full

Stronger memory

Use wiki, decisions, planning, verification, automated checks, and stronger handoff rules for larger or long-running projects.

The installer also supports practical profiles such as adopt, refresh, and custom. The profile is a starting point, not a badge of maturity.

Isolated experiments

Experiments are valuable, but they should not change the stable project before they are evaluated. POM treats experiments as temporary work until someone deliberately chooses what to keep.

  • Use a branch or Git worktree such as exp/<topic> for broad or risky work.
  • Use /tmp or experiments/<topic> for short trials and disposable evidence.
  • Keep trial dependencies, environment files, generated output, and external repositories isolated.
  • Promote only selected learning: discard, synthesize, write a decision, create a task plan, or reimplement cleanly.

Git and POM

Git is the detailed historical memory. It records what changed and when. POM is the current operating synthesis. It explains what matters now, why it matters, and where to resume.

Do not turn POM documents into manual changelogs. If historical detail is needed, read Git. If restart context is needed, read POM.

Final recommendations

  • Start with the smallest useful POM profile.
  • Create memory only when it helps future work restart, verify, or avoid rediscovery.
  • Read current sources before making claims about the project.
  • Keep desiderata and unresolved alternatives in Open Discussion until they are promoted to a spec, decision, wiki update, or task plan.
  • Check the Artifact Policy before editing files with special rules.
  • Use Git for history and POM for current operating synthesis.
  • Keep experiments isolated until their learning is selected for promotion.
  • Remove rules that add process without protecting memory.
  • When using or evolving POM itself, ask what a future reader or agent will need to know to take the next safe step in the project.
